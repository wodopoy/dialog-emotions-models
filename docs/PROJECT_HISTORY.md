# История проекта `dialog-emo-models` — полный журнал

Подробная хронология работы: что пробовали, что решали, полные таблицы
промежуточных экспериментов. Этот файл — нарратив (что и зачем делали по порядку);
сжатые результаты по итерациям дублируются в `RESEARCH_NOTES.md`.

---

## 0. Контекст и цель

Дипломный проект: **оффлайн, на устройстве** классификация эмоций коротких русских
диалоговых сообщений (клавиатура / «эмоциональный таймлайн» переписки). Ограничения:
маленький размер, быстрый инференс, без отправки данных наружу.

Стартовая ситуация: проект был разнесён по нескольким репозиториям —
- `undergradThesis` (текст диплома + всё подряд, GitHub: wodopoy),
- `dialog-emo-models` (обучение моделей, GitHub: NotHotTryHard),
- `dialog-emo-demo-codex` (Gradio-демо/визуализатор).

В `undergradThesis` были модели, которых не было в `dialog-emo-models`, прогнанные
в других условиях. **Решение:** свести все модели в `dialog-emo-models` под единый
интерфейс `EmotionModel.predict_logits(texts) -> (n, EMOTIONS)`, чтобы их можно было
обучать на общих данных и сравнивать в одном лидерборде.

---

## 1. Консолидация моделей (миграция из undergradThesis)

Перенесли **код моделей** под контракт `dialog-emo-models` (а не устаревшие бинарники —
они были обучены на другом датасете/схеме и несравнимы):

- `lexicon` — словарный baseline (русские стемы → 6 эмоций).
- `word+char TF-IDF` (union) — тезисный `tfidf_lr`, добавлен как опция к ridge/logreg.
- `fastText supervised` — нативный fastText, сделан joblib-сериализуемым.
- `rubert-tiny2 fine-tune` (+ONNX-экспорт — позже **удалён** как «та же модель, не нужен»).

Что **не** переносили: `char_ngram_linear` (= уже существующий char-tfidf), `fasttext_proxy`
(sklearn-затычка), `1stmodel` (прототип с другой схемой) — обоснованно дублирующие/легаси.

Каждый перенос — отдельный коммит в стиле репозитория («Add …», императив, без тела).
Приватность переписок не трогали — диалоги синтетические.

**Найдено по дороге:** деплой-победитель по KL — `logreg-tfidf`, а не `ridge` (README с
«ridge recommended» был перевёрнут относительно данных); `rubert-tiny2` ≈ 29M параметров /
~117 МБ (не 11.8M — путали со старым rubert-tiny).

## 2. Стратегический research-пасс (13 агентов)

Перед прогонами провели многоагентный research по 6 направлениям (пригодность моделей,
датасеты, квантизация, обучение лёгких, валидация, пробелы) с адверсариальной проверкой
фактов. Ключевые выводы (всё подтверждено по артефактам репо):

- **logreg ≫ ridge по KL/калибровке** (logreg-tfidf KL ≈ 1.017, ridge ≈ 1.639; ridge
  переуверен conf 0.88 @ acc 0.62).
- **rubert-tiny2 ≈ 29M / 117 МБ**, ~92% весов — эмбеддинги → `torch.quantize_dynamic`
  (только Linear) бесполезен; нужен ONNX INT8. **Квантизацию отложили.**
- **Главный пробел** — нет деплой-метрик (latency/size/RAM/ECE), хотя весь смысл «маленький
  и быстрый на телефоне». → построили харнесс деплой-метрик.
- `word+char union` и `fastText` исторически не были померяны; fastText дефолтный — слаб.
- Датасеты: оставить RuGoEmotions как primary; CEDR/XED — кандидаты на кросс-домен.

---

## 3. Инфраструктура бенчмарка

Построили под единый, сопоставимый прогон:
- `metrics.py` — единый набор: top-1 acc, macro/weighted/micro-F1, MAE/MSE, **KL, JS**,
  **ECE**, per-class F1, матрица 6×6 + **деплой-метрики** (size, load, latency p50/p95 @ batch=1).
- **Полы:** `prior` (распределение классов трейна — честный вероятностный пол) и `majority`.
- **Унификация препроцессинга:** все модели через `normalize_text` (эмодзи → слова-цели; не
  затираем полезное).
- Данные: `seara/ru_go_emotions` (config simplified), официальный сплит **80/10/10**
  (train 39055 / val 4905 / test 4870 после маппинга 28→6 групп).

**Метрика отбора — KL** (мягкие метки = распределение; юзкейс — таймлайн, где важно само
распределение; KL = обобщённая кросс-энтропия для soft-таргетов). Рядом всегда отчитываются
accuracy/macro-F1/ECE/JS. Аргументы для защиты — в `RESEARCH_NOTES.md`.

## 5. Эволюция схемы меток (6 → 7 классов)

CEDR ложился в схему только на 4 из 6 эмоций. Разобрали группировку: 6 меток были
прикладным огрублением GoEmotions-28, но `warmth` — кастомная категория (нет нигде), а
`surprise` (есть в CEDR) был выкинут. Чтобы **CEDR ложился целиком**, добавили класс
**`surprise`** (туда же realization/curiosity/confusion из выкинутого) + вернули relief→joy.
Итог — **7 классов:** `joy, warmth, sadness, anger, anxiety, surprise, neutral`. Схема
переключается env-переменной `EMO_SCHEME` (дефолт 6 не тронут).

Эффект: восстановлены данные (rugo train 39055→43021), CEDR теперь покрывает 6/7 классов
(а не 4/6); `warmth` остаётся «GoEmotions-only» (в CEDR его нет — это не чинится).

## 6. Доменная адаптация (CEDR) — итерация 8

Вопрос: разрыв на нативном русском — это нехватка нативных данных или потолок модели?
Прогон: train на RuGo / RuGo+CEDR / только CEDR, тест на **обоих** (CEDR-test held out).
7-класс, фикс-параметры. Матрица (rugo_acc / cedr_acc):

| модель | A: RuGo | B: RuGo+CEDR | C: CEDR |
|---|---|---|---|
| logreg-char | 0.579 / 0.494 | 0.581 / **0.699** | 0.361 / 0.718 |
| logreg-union | 0.585 / 0.498 | 0.586 / **0.706** | 0.360 / 0.711 |
| nb-multinomial | 0.539 / 0.508 | 0.538 / 0.654 | 0.355 / 0.642 |
| fasttext | 0.482 / 0.385 | 0.477 / 0.622 | 0.321 / 0.713 |
| rubert-tiny2-finetune | 0.617 / 0.594 | 0.616 / **0.776** | 0.440 / 0.801 |
| deberta (frozen) | 0.638 / 0.647 | — | — |

**Ключевой результат:** режим **B (RuGo+CEDR) даёт +18-21 п.п. на CEDR при НУЛЕВОЙ потере
на RuGo**, от ~7.5k нативных примеров. То есть разрыв был **в основном от нехватки нативных
данных, а не ёмкости модели:** linear с нативными данными (0.699) обходит замороженный
deberta (0.647). Режим C (только CEDR) роняет RuGo (0.36) — нужен микс. Практический вывод:
собрать чуть-чуть размеченных данных целевого домена и до-обучать лёгкую модель — дешевле и
эффективнее, чем тащить трансформер на телефон.

(Оговорки: CEDR это LiveJournal/новости/Twitter, **не диалог**; `warmth` от CEDR не учится.)

## 8. Журнал ключевых решений

- **Переносим код моделей, не бинарники** (бинарники с другого датасета несравнимы).
- **Метрика отбора — KL** (мягкие метки + таймлайн); рядом acc/F1/ECE/JS.
- **ONNX-экспорт удалён** из rubert — «та же модель», деплой-формат, не нужен в сравнении.
- **Квантизация трансформеров — отложена** (гейтится тяжёлым обучением; не приоритет).
- **6 → 7 классов** ради совместимости с CEDR (+surprise).
- **Деревья — не деплой-кандидаты** (dominated + 119-928 МБ), оставлены как «пробовали».
- **Не брать `*ru_goemotions`-варианты** (Djacon и пр.) — тот же Reddit-источник → утечка.
- **Деплой-выбор — `logreg-char`** (лучший размер/качество/калибровка in-domain и приемлемо
  кросс-домен; с до-обучением на нативных данных почти догоняет трансформер).

## 9. Операционные заметки (грабли и параллелизм)

- **fastText + NumPy 2:** обёртка `fasttext-wheel` падает на `np.array(copy=False)` → зовём
  C++-предиктор напрямую (`model.f.predict`). Собирается из исходников на Python 3.14.
- **Параллельный HPO memory-bound:** TF-IDF на 49k — упор в RAM, не CPU. На M4 Pro (14 ядер,
  48 ГБ) **14 воркеров уходят в своп** и медленнее, чем 8-10. Оптимум ~8-10 без свопа.
- **`pkill -f <script>.py` НЕ убивает spawn-воркеров** (их cmdline `multiprocessing.spawn`) —
  глушить по `pkill -f .venv/bin/python`, иначе остаются осиротевшие процессы.
- HPO-скрипты не сохраняли веса (дампили для замера размера и удаляли) — финальные модели
  пересохранены отдельно.

## 10а. Температурная калибровка (по ECE) — итерация 10

Пост-хок temperature scaling всех 20 чекпоинтов: один скаляр T делит логиты до softmax,
argmax инвариантен → accuracy/F1 не меняются (калибровка бесплатна). T подобран per-model на
val RuGo по ECE (контроль по NLL — совпадает) с дедбандом ε = 0.005, «до/после» — на held-out
RuGo test + CEDR; reliability diagrams — `docs/img/calibration/`. Полный разбор —
`docs/CALIBRATION.md`, данные — `artifacts/experiments/calibration/`.

- **Переуверенные лечатся бесплатно:** семейство `ridge` ECE 0.26 → 0.03, пресет `maxkazak`
  0.39 → 0.01 in-domain (T ≈ 2.3–4.0), accuracy без изменений.
- **Деплой `logreg-char` калиброван из коробки** (ECE 0.028) — дедбанд держит T = 1.
- **Дедбанд ε = 0.005:** без него у уже калиброванных моделей (`logreg-char`, `fasttext`,
  `tree-hgb`) RuGo ECE чуть *рос* — val-минимум по ECE это сэмплинг-шум, не переносится на test
  (T(ECE) = 0.98 против T(NLL) = 1.03–1.42, разнонаправленно). Порог лёг в фактический ~6× разрыв
  между переносимыми и шумовыми выигрышами по val ECE.
- **Калибровка доменно-зависима:** in-domain T переносится на CEDR у переуверенных, но у уже
  калиброванных (`fasttext`, `rubert-finetune`) RuGo-T CEDR не лечит/ухудшает; нативный dev
  (oracle T) закрывает разрыв (`fasttext` CEDR 0.110 → 0.020).
- **Полная матрица dev×test (`scripts/calibrate_combined.py`):** T на {rugo_val, cedr_val,
  rugo_val+cedr_val} оценивается на {rugo_test, cedr_test, совокупность}, все 20 моделей
  (cedr_val = половина CEDR test, cedr_test = вторая, непересекающиеся). Закономерность:
  **лучший dev = домен оценки.** `T_cedrval` сильно чинит CEDR (`fasttext` 0.123 → 0.026,
  `deberta` 0.104 → 0.021, `logreg-char` 0.067 → 0.033), `T_comb` — лучший компромисс на
  совокупности (но RuGo-перекошен ~85%, на CEDR ≈ T_rugo). Один скаляр не оптимален на всех трёх →
  для нативного деплоя калибровать на cedr_val, под смешанный вход — T_comb.
- **ECE-fit vs NLL-fit (`scripts/calibrate_objective_comparison.py`, `docs/CALIBRATION_OBJECTIVE.md`):**
  отбор T по ECE и по NLL, репорт ECE+KL на rugo/cedr test. In-domain ECE → ECE-fit (14/17);
  **деплой-домен CEDR ECE → NLL-fit (12/17); KL → NLL-fit (14/17)**. NLL — гладкий proper scoring
  rule, лучше переносится кросс-домен; патология — fasttext/maxkazak (ненастоящие логиты,
  NLL пересмягчает). **Рекомендация: оба исследования в работу, продакшн — NLL-fit +
  ECE-предохранитель** (откат к T=1, если val ECE хуже raw).

Примитивы: `apply_temperature` / `best_temperature(objective, min_improve)` /
`negative_log_likelihood` / `reliability_curve` в `metrics.py`, тесты —
`tests/test_calibration.py`, прогон — `scripts/calibrate_temperature.py`, картинки —
`scripts/plot_reliability.py`.

## 12. Итоговый нарратив для диплома

Полный спектр моделей (полы → лексиконы → linear → fastText → NB → деревья → трансформеры)
сравнён на одном протоколе, с HPO, деплой-метриками, анализом ошибок, learning curve,
кросс-доменом и доменной адаптацией.

Главные тезисы:
1. **In-domain лёгкая `logreg-char` (2.7 МБ) почти догоняет трансформер** по качеству и
   **лучше всех калибрована** — идеальна для оффлайн-клавиатуры.
2. **На смене домена линейные проседают ~10-15 п.п.**, трансформеры — нет: in-domain сила
   линейных частично = переобучение под translationese.
3. **Но это лечится дёшево:** ~7.5k нативных примеров (режим RuGo+CEDR) поднимают линейную
   на CEDR на +20 п.п. без потери на RuGo, почти до уровня трансформера. → собрать немного
   in-domain данных важнее, чем менять модель.
4. Деревья — мимо (dominated + гигантские), NB — приличная классика, fastText с правильными
   параметрами/температурой — быстрый и достойный.

Финальная рекомендация: **`logreg-char`, обученная на RuGo+CEDR (а в идеале — с дотюном на
размеченных диалогах целевого домена).**

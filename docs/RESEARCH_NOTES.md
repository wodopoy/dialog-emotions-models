# Research notes / журнал экспериментов

Рабочий файл со всеми нюансами обучения и проведёнными исследованиями — чтобы
потом ничего не забыть при написании диплома. Пополняется по ходу работы.

## Задача и схема меток

- Цель: **оффлайн, на устройстве** классификация эмоций коротких русских
  диалоговых сообщений (клавиатура / «эмоциональный таймлайн» переписки).
  Ограничения: маленький размер, быстрый инференс, без отправки данных наружу.
- Схема: **6 эмоций** — `joy, warmth(забота), sadness, anger, anxiety, neutral`.
  Шесть меток выбраны осознанно: (1) упростить задачу для лёгких моделей,
  (2) выделить основные эмоции. Это не попытка покрыть весь спектр — это
  проектное упрощение, и так и описываем в работе.
- Метки **мягкие** (распределение вероятностей на сообщение): получены из
  RuGoEmotions маппингом 28 GoEmotions-классов в 6 групп; мультилейбловые строки
  делят массу поровну между активными группами (≈8.5% обучающих строк
  мультигрупповые, остальные ~91.5% — one-hot).

## Данные

- Primary train/val/test: `seara/ru_go_emotions` (config `simplified`, MIT) —
  RuGoEmotions = машинный перевод (Google) английского GoEmotions (Reddit) на
  русский. После маппинга в 6 групп и фильтрации немаппящихся строк:
  **train 39055 / val 4905 / test 4870** (raw 43410/5426/5427).
- Известное ограничение: домен (Reddit-комментарии) и перевод ≠ телефонный
  диалог. Кросс-доменные датасеты (XED-RU, CEDR) пока решено **не** подключать.

## Модели (под единым контрактом `EmotionModel.predict_logits(texts)->(n,6)`)

Лёгкие / деплоимые:
- `lexicon` — словарь стем-слов (без обучения), пол-baseline.
- `ridge-tfidf`, `logreg-tfidf` — char_wb(3-5) TF-IDF + Ridge / LogReg.
- `ridge-word-tfidf`, `logreg-word-tfidf` — word(1-2) TF-IDF.
- `ridge-word-char-tfidf`, `logreg-word-char-tfidf` — word+char FeatureUnion.
- `fasttext-supervised` — нативный fastText (subword), quant `.ftz`.
- baselines: `majority` (всегда мода = neutral), `prior` (частоты классов трейна).

Тяжёлые (потолки качества, не деплоятся как есть):
- `rubert-tiny2-finetune` (база `cointegrated/rubert-tiny2`).
- HF-пресеты: `seara/...`, `fyaronskiy/deberta...`, `MaxKazak/ruBert-base...`.

## Ключевые решения и нюансы

- **Деплой-победитель по headline-метрике (soft KL) — `logreg-tfidf`, НЕ `ridge`.**
  По данным репо: logreg-tfidf val KL ≈ 1.017, ridge-tfidf ≈ 1.639; ridge сильно
  переуверен (confidence ≈ 0.88 при accuracy ≈ 0.62), logreg калиброван заметно
  лучше (≈0.54 @ 0.61). Прежняя формулировка «ridge — основной baseline» в README
  была перевёрнута относительно данных.
- **`rubert-tiny2` ≈ 29M параметров / ~117 МБ** (не 11.8M — то старый rubert-tiny).
  ~92% весов — таблица эмбеддингов, поэтому `torch.quantize_dynamic` (только Linear)
  жмёт ~6%. Деплой трансформера на мобилу требует ONNX Runtime INT8 — **отложено**.
- **fastText + NumPy 2:** `fasttext-wheel` 0.9.2 в Python-обёртке делает
  `np.array(probs, copy=False)`, что **падает на NumPy ≥ 2**. Обходим вызовом
  C++-предиктора напрямую: `model.f.predict(text, k, 0.0, "strict")` (см.
  `models/fasttext.py`). Собирается из исходников на Python 3.14 нормально.
- **Унификация препроцессинга:** раньше sklearn-линейные модели НЕ звали
  `normalize_text`, а lexicon/fastText/transformer — звали. Сделали единый прогон
  всех через `normalize_text`. Важно: `normalize_text` **парсит эмодзи в слова-цели**
  (😢→«грусть» и т.п.) и не выкидывает полезную информацию — способность
  использовать эмодзи считаем плюсом, а не шумом.
- **maxkazak (KL 5.165) — не «сломан», а несовместим со схемой меток** (обучен на
  9-лейбловой Izard-схеме Djacon; харнесс агрегирует по именам GoEmotions). Из
  основного сравнения исключаем (или чиним маппинг отдельно).
- **`*-word-char-tfidf` union и `fasttext` исторически не были померяны** — в этой
  итерации добавлены в прогон.

## Отложено / решено не делать (пока)

- Квантизация трансформеров (ONNX INT8) — гейтится обучением rubert; отложено.
- Кросс-доменные датасеты (XED-RU, CEDR), silver-label дистилляция — не сейчас.
- Калибровка (temperature/ECE как отдельный шаг отчёта) — позже.
- soft-vs-hard ablation, прогон на реальном телефоне, секция limitations — позже.

### Итерация 1 — лёгкие модели, дефолтные гиперпараметры

Прогон: `python scripts/evaluate_ru_go_emotions.py` (train 39055, val 4905, test 4870).
Метрики на **validation**, сортировка по KL (меньше — лучше). Гиперпараметры —
дефолтные (HPO ещё не делали).

| model | acc | macro-F1 | KL | JS | MAE | ECE | size MB | p50/p95 ms |
|---|---|---|---|---|---|---|---|---|
| logreg-word-char-tfidf | 0.626 | 0.587 | **0.969** | 0.254 | 0.174 | 0.075 | 37.5 | 1.83 / 1.93 |
| logreg-tfidf (char) | 0.614 | 0.575 | 1.017 | 0.268 | 0.182 | 0.087 | 4.1 | 0.30 / 0.39 |

Выводы итерации 1:
- **Победитель — `logreg-word-char-tfidf`** по accuracy (0.626), macro-F1 (0.587) И KL
  (0.969). Union (word+char) бьёт char-only — т.е. словный канал НЕ лишний (раньше
  union ни разу не мерили). Но цена: 37.5 МБ и 1.8 ms против 4.1 МБ / 0.3 ms у char-only
  `logreg-tfidf` (у union-векторайзера нет `max_features` cap → огромный словарь).
  Баланс размер/качество лучше у `logreg-tfidf`; union даёт +1.2 п.п. acc / −0.05 KL
  за ~9× размер. **HPO с `max_features` на union — следующий шаг.**
- `logreg-tfidf` воспроизводит ровно прежние цифры (0.614 / 1.017) — харнесс
  консистентен, плюс теперь есть baselines, union, fastText и деплой-метрики.
- **logreg >> ridge по KL/ECE.** Ridge даёт лучший MAE (0.129), но KL 1.64 и ECE 0.26
  (сильно переуверен). Для таймлайна (нужны калиброванные вероятности) — logreg.
- **fastText (дефолт softmax) слаб по вероятностям:** KL 3.21 и ECE 0.39 — хуже
  prior-floor, хотя он самый быстрый (0.024 ms) и маленький. argmax-обучение + softmax
  → пиковые переуверенные распределения. Пробовать `loss=ova` и HPO; как есть —
  кандидат только по скорости, не по качеству.
- **Полы работают:** `prior` KL 1.48 — честный вероятностный пол (uniform `dummy`
  1.73 — не пол); `majority` KL 18 показывает, что one-hot предсказание катастрофично
  на мягких метках.
- **Эффект унификации препроцессинга на RuGoEmotions ≈ ноль** (logreg-tfidf не
  изменился): эмодзи в переведённом Reddit почти нет. Это важно для синтетических
  диалогов с эмодзи, не для этого датасета.
- Деплой-метрики (size / latency @ batch=1) теперь есть для всех; все лёгкие модели
  на порядки меньше/быстрее трансформера (~117 МБ).

Артефакты: `artifacts/experiments/light-leaderboard/` (gitignored).

### Итерация 2 — HPO лёгких моделей (66 конфигов)

Прогон: `python scripts/tune_light_models.py` (отбор по **val KL**, победители
оценены на тесте). Лучшие по семействам:

| family | val KL | val acc | val macroF1 | val ECE | test KL | test acc | size MB | p50/p95 ms | лучшие параметры |
|---|---|---|---|---|---|---|---|---|---|
| **logreg-union** | **0.916** | **0.643** | 0.571 | 0.027 | 0.921 | 0.633 | 5.71 | 0.55/0.64 | word+char, max_features=20k, C=1, class_weight=None, sublinear_tf |
| **logreg-char** | 0.932 | 0.636 | 0.570 | 0.026 | 0.939 | 0.625 | **2.46** | **0.22**/0.27 | char_wb(3-5), max_features=30k, C=2, class_weight=None, sublinear_tf |
| logreg-word | 1.001 | 0.613 | 0.510 | 0.053 | 1.009 | 0.602 | 2.74 | 0.19 | word(1-2), 30k, C=1, cw=None |
| ridge-char | 1.440 | 0.630 | 0.560 | 0.244 | 1.443 | 0.636 | 8.20 | 0.44 | char_wb(3-5), 100k, alpha=3 |
| prior | 1.479 | 0.332 | 0.083 | 0.018 | — | — | 0 | — | floor |
| ridge-union | 1.505 | 0.632 | 0.567 | 0.250 | 1.499 | 0.634 | 12.7 | 0.79 | 100k, alpha=3 |
| lexicon | 1.526 | 0.406 | 0.272 | 0.071 | — | — | 0.001 | 0.01 | — |
| fasttext | 2.683 | 0.518 | 0.473 | 0.170 | 2.655 | 0.507 | 5.76 | **0.023** | lr=0.3, epoch=25, loss=**ova** |
| majority | 18.0 | 0.332 | 0.083 | 0.668 | — | — | 0 | — | floor |

Главные выводы (всё подтверждено перебором, годится в работу):
- **HPO заметно улучшил logreg.** char-logreg: KL 1.017→**0.932**, ECE 0.087→**0.026**.
  Решающие гиперпараметры: **`class_weight=None`** (захардкоженный `balanced` искажал
  вероятности/калибровку — синтез это и предсказывал) + **`sublinear_tf=True`** + `C` 1→2.
- **logreg ≫ ridge окончательно.** Лучшие logreg: ECE ≈ 0.026 (отличная калибровка) и
  KL ≈ 0.92. Лучшие ridge: ECE ≈ 0.24 (сильно переуверен), KL ≈ 1.44 — несмотря на
  лучший MAE (0.125). Для таймлайна (нужны калиброванные вероятности) — однозначно logreg.
- **Размер: меньше = лучше.** Для union `max_features=20k` дал и лучший KL (0.916), и
  наименьший размер (5.7 МБ) — против 50k/100k (8.3/12.7 МБ, KL чуть хуже). Прежние
  37 МБ были артефактом отсутствия cap. char-logreg на 30k — всего **2.46 МБ / 0.22 ms**.
- **fastText: ova > softmax** (KL 2.68 vs 3.07; ECE 0.17 vs 0.39), но по качеству всё
  равно слаб — хуже prior-пола по KL, acc 0.52 (ниже linear 0.64). Остаётся кандидатом
  только по скорости (0.023 ms) — не по качеству.
- **Заголовок результата:** тюнингованный линейный `logreg-union` (val acc 0.643 / KL 0.916,
  5.7 МБ) **обходит fine-tune RuBERT-tiny2** (0.630 / 0.898, ~117 МБ) по accuracy и
  приближается к потолку DeBERTa (0.680 / 0.883, ~496 МБ) — при размере в 20-90× меньше.
  (Цифры тяжёлых — из прежнего `validation-full-report`; нужно переснять их тем же
  харнессом для строго одинакового протокола — следующий шаг.)

Рекомендация по деплою: **`logreg-char` (max_features=30k)** как дефолт (2.46 МБ, 0.22 ms,
KL 0.932, ECE 0.026); **`logreg-union` (20k)** как «макс. качество» (+1 п.п. acc / −0.016 KL
за ~2.3× размер). Артефакты: `artifacts/experiments/light-tuning/` (`best__*.joblib`).

Выводы (главные для работы):
- **Свежий `rubert-tiny2-finetune` — лидер по KL (0.842)**, обходит DeBERTa (0.883) при
  размере в 4x меньше. Наш дообученный лучше прежних 0.630/0.898 из validation-full-report
  (2 эпохи / argmax / max_len 96) — это наша воспроизводимая цифра.
- **DeBERTa — лидер по accuracy (0.680) и macroF1 (0.641)**, но 475 МБ, p95 268 ms и
  слабая калибровка (ECE 0.132).
- **Лёгкие linear — лучшая калибровка из ВСЕХ** (logreg-char/union ECE 0.026/0.027 < всех
  трансформеров). Для таймлайна (нужны калиброванные вероятности) — решающий аргумент.
- **Разрыв деплой--потолок мал:** `logreg-char` (2.46 МБ, 0.22 ms) — acc 0.636 / KL 0.932:
  это ~ -4.4 п.п. acc к DeBERTa (0.680) при размере в 193x меньше и латентности в ~1000x ниже;
  по KL близко к DeBERTa, по ECE — лучше неё.
- **HF-пресеты воспроизвели прежние цифры точно** (deberta 0.680, seara 0.559, maxkazak 0.515 /
  KL 5.17) → протокол консистентен, сравнение лёгких и тяжёлых строго одинаковое.
- `maxkazak` KL 5.17 — снова артефакт несовпадения схемы меток (исключаем из честного сравнения).
- p95 трансформеров шумный (cold-start на desktop); p50 чище и показывает разрыв 100-1000x с linear.

Нарратив для работы: на Pareto качество--размер linear держат угол маленький+быстрый+
калиброванный почти вровень по качеству; трансформеры покупают +неск. п.п. accuracy ценой
50-200x размера и 100-1000x латентности. Выбор для оффлайн-клавиатуры: `logreg-char` (деплой) /
`rubert-tiny2-finetune` (если позволяет бюджет — лучший KL).

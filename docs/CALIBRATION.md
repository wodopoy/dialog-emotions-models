# Температурная калибровка моделей (по ECE)

Раздел для диплома. Лидерборды репортят «сырую» (T=1) калибровку каждой модели; здесь —
пост-хок калибровка температурой всех 20 сохранённых чекпоинтов и честное сравнение
**ECE до/после**. Источник: `scripts/calibrate_temperature.py` →
`artifacts/experiments/calibration/temperature_calibration.csv`.

## Зачем

**ECE (Expected Calibration Error)** — top-label калибровка: среднее по бинам уверенности
`|средняя уверенность − фактическая точность|`. ECE = 0 означает, что когда модель говорит
«0.8», она права в 80% случаев. Для шкалы эмоций на таймлайне это важно: высота столбика
должна отражать реальную вероятность, а не «модель всегда орёт 0.99».

Беда линейных TF-IDF моделей (наши деплой-кандидаты) — **переуверенность**: их softmax
почти one-hot, тогда как top-1 accuracy ≈ 0.5–0.7, поэтому top-label ECE = 0.25–0.40.

## Метод

**Temperature scaling** (Guo et al., 2017): один скаляр T делит логиты *до* softmax —
`softmax(logits / T)`. `T>1` смягчает распределение (снижает уверенность), `T<1` заостряет,
`T=1` — no-op. Применяется единообразно к любой модели через её `predict_logits`.

Ключевое свойство: **argmax инвариантен** при любом положительном T, поэтому
**accuracy / F1 / матрица ошибок не меняются** — калибровка «бесплатна» по качеству
классификации; двигаются только вероятности (KL, NLL, ECE).

Протокол (честный):

1. T подбирается **per-model на валидации RuGo** (отложенной от обучения), минимизацией
   **val ECE** — метрика *отбора*; но репортинг строго на held-out (шаг 4), а дедбанд (шаг 2)
   запрещает гнаться за её шумом, так что «оптимизируем ту же метрику, что репортим» здесь не
   работает как возражение.
2. **Дедбанд:** T остаётся ровно `1.0`, если выигрыш по val ECE меньше порога **ε = 0.005**.
   Binned ECE — шумная оценка на конечной выборке (бутстрап-SE на RuGo test ≈ 0.005, см. ниже),
   и у уже калиброванной модели её минимум на val лежит на волосок ниже T = 1 чисто по
   сэмплинг-шуму; дедбанд (≈ один SE) не даёт «поймать» этот шум.
3. **Контроль:** второй T подбирается по **NLL** (гладкий учебниковый суррогат, который
   классически минимизируют). T(ECE) ≈ T(NLL) у всех моделей с **реальной** коррекцией; у уже
   калиброванного хвоста критерии расходятся в направлении — это сам по себе признак, что
   сигнала нет (см. ниже), а не контрпример. Отбор по ECE не «подгоняет бины».
4. «До/после» считается на **held-out**: RuGo **test** (5371) + нативный **CEDR** test (1882).
   Температура подобрана не на этих данных → цифры не самооптимизированы.
5. Для CEDR добавлена колонка **oracle T** (подобран **на самом CEDR test, in-sample**) —
   оптимистичная **нижняя граница** ECE (по той же логике, что binned-минимум смещён вниз).
   Реальный нативный dev-набор (фит на dev, репорт на test) дал бы цифру **между** колонкой
   in-domain-T и oracle, а не на уровне oracle.

Всё считается на кэшированных логитах, так что подбор T — это десяток softmax на модель;
единственная реальная стоимость — однократная загрузка каждого чекпоинта.

## Выводы

1. **Где калибровка решает.** Самые переуверенные — пресет `maxkazak` (ECE 0.39) и всё
   семейство `ridge` (~0.26): softmax почти one-hot при accuracy 0.5–0.7. Температура
   (T ≈ 2.3–4.0) обрушивает их in-domain ECE до **0.01–0.04 без изменения accuracy**. Это и есть
   главный результат раздела: переуверенность лечится одним скаляром бесплатно. **Зеркальный
   случай — недоуверенные модели** (`nb-complement` 0.69, `tree-rf` 0.68, `lexicon-hand` 0.78):
   там T < 1 заостряет размазанный softmax и так же снижает ECE (`tree-rf` 0.094 → 0.018).
   Температура лечит обе стороны — переуверенность (T > 1) и недоуверенность (T < 1).
2. **Кому не нужна.** `logreg-char`, `tree-hgb`, `fasttext` уже хорошо калиброваны in-domain
   (ECE 0.016–0.028) — дедбанд держит их на T = 1 (raw = cal). То есть рекомендуемый деплой
   `logreg-char` достоверен «из коробки», калибровать его не нужно.
3. **Отбор по ECE корректен.** Для всех моделей с **реальной** коррекцией T(ECE) ≈ T(NLL) —
   совпадение с учебниковым NLL-критерием показывает, что отбор по ECE не эксплуатирует бины.
   Расходятся два рода случаев: (а) `maxkazak` (NLL упёрся в потолок сетки 20 — экстремальная
   переуверенность, ECE нашёл внутренний оптимум 4.0); (б) уже калиброванный хвост (`logreg-char`
   T(ECE)=0.98 против T(NLL)=1.03, `fasttext` 0.98 против 1.42) — но там расхождение направлений
   и есть доказательство, что калибровать нечего (см. следующий раздел), а дедбанд оставляет их
   на T = 1.
4. **Калибровка доменно-зависима.** In-domain T переносится на CEDR и снижает ECE у
   переуверенных (`ridge` 0.19 → 0.05; `maxkazak` 0.235 → 0.164). Но для уже калиброванных
   in-domain моделей RuGo-температура CEDR не лечит (`fasttext` остаётся на T = 1 и CEDR 0.110),
   а у `rubert-finetune` навязанный T даже ухудшает CEDR (0.026 → 0.076). Колонка **oracle T**
   показывает разрыв: нативный dev целевого домена закрывает его почти полностью (`fasttext`
   CEDR 0.110 → 0.020, `maxkazak` 0.235 → 0.057). Вывод: калибровать надо на dev той области,
   где вероятности будут использоваться.
5. **Практика деплоя.** Если выбрать `ridge-char` ради более высокой CEDR-accuracy (0.68), его
   **обязательно** калибровать (T ≈ 2.3) — иначе шкала уверенности на таймлайне недостоверна
   (ECE 0.26). `logreg-char` калибровки не требует.

## Почему ECE мог вырасти после калибровки (и почему в финале не растёт)

Если подбирать T «наивно» (минимум val ECE без дедбанда), у уже хорошо калиброванных моделей
(`logreg-char`, `fasttext`, `tree-hgb`) RuGo ECE слегка **росло** после калибровки. Причина —
переобучение под шум бинированного ECE, а не реальная калибровка.

Эти модели уже у «пола» ECE in-domain (0.016–0.028). Сам **10-бинный ECE — шумная оценка**:
бутстрап по RuGo test даёт SE ≈ 0.005 (logreg-char ECE 0.028 ± 0.005, ridge-char 0.257 ± 0.006).
То есть улучшение порядка 0.001–0.002 неотличимо от нуля. Минимум val ECE лежит на ~0.001–0.002
ниже T = 1 чисто случайно, и этот «выигрыш» не переносится на test (все цифры — колонки
`val_ece_raw`, `val_ece_free`, `test_ece_free` в CSV, T\* без дедбанда):

| модель | val ECE @T=1 | val ECE @T\* | test ECE @T=1 | test ECE @T\* | T\*(ECE) | T\*(NLL) |
|---|---|---|---|---|---|---|
| logreg-char | 0.0241 | **0.0224** | 0.0284 | 0.0320 | 0.98 | 1.03 |
| fasttext | 0.0154 | **0.0140** | 0.0157 | 0.0169 | 0.98 | 1.42 |

T = 0.98 минимизирует **val** ECE, но на **test** оптимум наоборот — T > 1; то есть val-минимум
сидит не с той стороны от единицы, и навязанный T поднимает test ECE на ~0.001–0.004 (внутри
SE). Здесь T\* = 0.98 — **не** артефакт сетки: точка T = 1.0 явно входит в сетку перебора, и
0.98 всё равно выигрывает на val (на 0.0017) — значит это настоящий, пусть и шумовой, под-единичный
минимум. Два независимых признака, что сигнала нет:

- **направления критериев расходятся:** T(ECE) = 0.98 (заострить) против T(NLL) = 1.03–1.42
  (смягчить). У калиброванной модели согласованного направления нет — критерии бродят по шуму;
- **знак переноса:** по всем 17 моделям выигрыш по val ECE (`val_gain_free`) > ~0.005 ⇔ выигрыш
  и на test (перенос есть); < 0.002 ⇔ на test ECE падает (шум). Между группами чистый разрыв
  ~6.5× (ближайший «реальный» — `logreg-union`, val-выигрыш 0.011, test 0.042 → 0.022; «шумовые»
  — 0.0012–0.0017, test растёт).

Поэтому в отборе стоит **дедбанд ε = 0.005 (≈ один SE ECE)**: он ровно по этому разрыву отделяет
реальные коррекции (которые переносятся на test) от шумовых. С ним `logreg-char` / `fasttext` /
`tree-hgb` остаются на T = 1 (raw = cal, без роста), а большие выигрыши (`ridge` 0.26 → 0.03,
`maxkazak` 0.39 → 0.01) проходят как есть. Это та же причина, по которой температуру классически
подбирают по гладкому NLL, а ECE только репортят, — дедбанд даёт тот же эффект, не отказываясь
от ECE-критерия.

(На CEDR ECE после in-domain-T у части моделей всё же растёт — но это уже не шум, а
доменный сдвиг: см. вывод 4 и следующий раздел.)

## Калибровать на RuGo, CEDR или совокупности? (полная матрица)

Законный вопрос: модели обучаются на **RuGo + CEDR**, а T подбирается на **RuGo val** — почему
не на совокупности? Прямой ответ — полная матрица: T фитится на трёх dev-наборах
(`rugo_val`, `cedr_val`, `rugo_val + cedr_val`) и оценивается на трёх test-наборах
(`rugo_test`, `cedr_test`, совокупность). У CEDR нет val-сплита, поэтому **`cedr_val` = половина
CEDR test, `cedr_test` = вторая половина** (непересекающиеся, обе held-out от весов). **Все три
dev-T — с тем же дедбандом ε = 0.005, что и основной пайплайн**, чтобы все колонки были в одной
политике (а не raw=дедбанд-вкл против @T=дедбанд-выкл); `raw` = T = 1. `T = 1.000` в строке T
означает, что дедбанд оставил модель как есть на этом dev → `@T = raw`.

**RuGo test** — ECE под каждым dev-T:

| модель | raw | T_rugoval | T_cedrval | T_comb |
|---|---|---|---|---|
| ridge-char | 0.257 | **0.025** | 0.040 | **0.025** |
| hf-fyaronskiy-deberta | 0.140 | **0.027** | 0.039 | 0.028 |
| rubert-tiny2-finetune | 0.054 | **0.013** | 0.054 | 0.023 |
| logreg-char | **0.028** | **0.028** | 0.078 | **0.028** |
| fasttext | **0.016** | **0.016** | 0.104 | 0.022 |

| модель | raw | T_rugoval | T_cedrval | T_comb |
|---|---|---|---|---|
| hf-maxkazak-rubert-base | 0.226 | 0.183 | **0.074** | 0.164 |
| ridge-char | 0.164 | 0.062 | **0.053** | 0.062 |

|---|---|---|---|---|
| ridge-char | 0.243 | **0.022** | 0.032 | **0.022** |
| hf-fyaronskiy-deberta | 0.134 | **0.027** | 0.033 | **0.027** |
| rubert-tiny2-finetune | 0.042 | 0.020 | 0.042 | **0.014** |
| logreg-word | 0.040 | 0.024 | 0.036 | **0.021** |
| logreg-char | **0.023** | **0.023** | 0.064 | **0.023** |

Закономерность чистая: **лучший dev = тот, что совпадает с доменом оценки.**

- **`T_rugoval`** — лучший на RuGo test (по построению); `T_cedrval` там стабильно худший. Где
  модель уже калибрована на RuGo (logreg-char, fasttext), дедбанд держит T = 1 → роста нет.
- **`T_cedrval`** — лучший на CEDR test, и выигрыш большой: fasttext 0.123 → **0.026**,
  deberta 0.104 → **0.021**, logreg-char 0.067 → **0.033**, maxkazak 0.226 → **0.074**.
- **`T_comb`** — лучший на совокупном тесте и почти-лучший на каждом домене по отдельности:
  **лучший единый компромисс**. Но он наследует RuGo-перекос (и dev, и совокупный test на ~85%
  RuGo: val 5374 против cedr_val 941), поэтому на нативном CEDR он близок к `T_rugoval`, а не к
  `T_cedrval`.
- **Ни один скаляр T не оптимален на всех трёх сразу:** `T_cedrval` жертвует RuGo (logreg-char
  RuGo 0.028, а под `T_cedrval` 0.078), `T_rugoval` — CEDR. Поэтому «просто калибровать на
  объединении» — не бесплатный обед: совокупность RuGo-доминирована, `T_comb` тянется к RuGo и
  недокалибровывает нативный домен.

Оговорка про `cedr_val`: он маленький (941), поэтому SE ECE на нём больше (~0.008), и дедбанд
ε = 0.005 (настроен под RuGo val, n = 5371) для него чуть слаб — у пары уже-CEDR-калиброванных
моделей (`logreg-word` 0.017 → 0.026, `nb-multinomial` 0.027 → 0.049) `T_cedrval` поднимает CEDR
ECE на шум. Это не баг политики, а следствие маленького целевого dev — ещё один довод за сбор
бóльшего нативного набора.

**Вывод для деплоя:** продукт работает на нативном русском → калибровать T на **целевом
(`cedr_val`-подобном) dev**. Нужна одна модель под смешанный вход → `T_comb`. RuGo-val в основном
прогоне — компромисс из-за отсутствия родного CEDR-val; матрица показывает, что́ даёт правильный
выбор. Идеал (повторяющийся вывод работы): собрать нативные **диалоговые** данные целевого домена
и калибровать (и дообучать) на них; при необходимости — **per-domain T**.

Воспроизвести (полный CSV со всеми 20 моделями и тремя test-доменами):
`python scripts/calibrate_combined.py` →
`artifacts/experiments/calibration/calibration_devset_comparison.csv`.

## Защита методики (anticipated questions)

- **Почему ECE, а не NLL для отбора T?** Прямое сравнение критериев
  (`docs/CALIBRATION_OBJECTIVE.md`) показало: для «нормальных» моделей разница в основном
  внутри 1–2·SE; ясный кросс-доменный (CEDR) выигрыш NLL — лишь у 3 моделей (tree-rf, deberta,
  lexicon-hand), а для деплой-модели `logreg-char` ECE-fit точечно лучше (NLL её даже ухудшает).
  NLL ломается на патологичных логитах (fasttext/maxkazak). **Вывод: продакшн остаётся ECE +
  дедбанд** (когерентно с репортируемыми числами, безопасно, лучше для деплой-модели); NLL —
  разобранная абляция, не замена.
- **Не оптимизируем ли мы метрику на тех же данных?** Нет: T подобран на val RuGo, а все
  репортируемые ECE — на held-out (RuGo test и CEDR). ECE с 10 равноширинных бинов — оценка с
  SE ≈ 0.005 (бутстрап), поэтому разделение fit/report принципиально.
- **Дедбанд ε = 0.005 — не подгонка ли?** Нет: порог не подбирался под «красивую» таблицу — он
  равен бутстрап-SE ECE (≈ 0.005) и попадает в фактический ~6× разрыв `val_gain_free` между
  переносимыми и шумовыми выигрышами (он же — признак переноса на test). Это не независимый
  свип по ε, а следствие того же разрыва: порог порядка 0.003–0.008 даёт ту же картину.
- **Почему accuracy не меняется?** Деление логитов на положительный скаляр монотонно и
  сохраняет argmax (проверено юнит-тестом `tests/test_calibration.py`).
- **Это калибровка «в обучении»?** Нет — это пост-хок: ни одна модель не обучалась с
  калибровочным членом. Температура применяется к уже обученным чекпоинтам на инференсе.

## Воспроизведение

```bash
python scripts/calibrate_temperature.py                  # все 20 чекпоинтов -> CSV + MD
python scripts/calibrate_temperature.py --skip-heavy      # только линейные/baseline (быстро)
python scripts/calibrate_temperature.py --min-improve 0   # без дедбанда (видно «рост» ECE)
python scripts/calibrate_combined.py                      # RuGo vs CEDR vs совокупность как dev-набор
python scripts/plot_reliability.py                        # диаграммы надёжности + overview -> docs/img/calibration/
```

Диагностика «роста ECE» и ~6× разрыв воспроизводимы прямо из CSV: колонки `T_ece_free`,
`val_ece_raw`, `val_ece_free`, `val_gain_free`, `test_ece_free` (фит T без дедбанда; сравни
`val_gain_free` с `rugo_ece_raw` − `test_ece_free`). Бутстрап-SE ECE — две строки на
`expected_calibration_error` с ресэмплингом индексов test.

Примитивы калибровки (`apply_temperature`, `best_temperature` с `objective=ece|nll` и
`min_improve`, `negative_log_likelihood`, `reliability_curve`) живут в
`src/dialog_emo_models/metrics.py` и покрыты `tests/test_calibration.py`. Картинки строит
`scripts/plot_reliability.py` (matplotlib, dev-зависимость).
